# SPEC — `knw` Facts Retrieval Service

Dokku app **`knw`** → https://knw.chenchik.me

Minimal RAG **retrieval** layer (no generation, no query rephrasing). Stores atomic facts, retrieves by semantic similarity + rerank. Bun + Hono + Postgres/pgvector, deployed on Dokku.

## Scope

In: store facts, embed, vector search, Voyage rerank, return ranked facts.
Out (explicitly): query rephrasing / multi-query / HyDE, LLM generation, document chunking/ingestion pipelines, auth UI.

## Stack

- Runtime: Bun
- HTTP: Hono
- DB: Postgres + pgvector via the `postgres` driver
- Embeddings + rerank: Voyage API direct REST, not via gateway — keeps private data one hop closer

## Data model

Dedicated schema `knw` for isolation inside the shared `db`:

```sql
CREATE EXTENSION IF NOT EXISTS vector;
CREATE SCHEMA IF NOT EXISTS knw;

CREATE TABLE IF NOT EXISTS knw.fact (
  id         UUID PRIMARY KEY,
  content    TEXT NOT NULL,
  embedding  VECTOR(1024) NOT NULL,   -- MUST match EMBED_DIM and chosen document model output dim
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX IF NOT EXISTS fact_embedding_hnsw
  ON knw.fact USING hnsw (embedding vector_cosine_ops);
```

Notes:
- UUIDs are generated by the application with `crypto.randomUUID()`.
- Do not require `pgcrypto` for id generation.
- No `metadata JSONB` in v1.
- Run schema setup idempotently on boot, following the `ensureSeededOnce` style from the persistence post.
- If an existing table’s vector dimension does not match `EMBED_DIM`, fail fast. Changing embedding dimension requires re-index/re-embed.

## Internal module boundary (the contract)

```ts
// The embedding model is a FIXED INVARIANT of stored vectors.
// Store + query must use compatible embedding spaces.
// Voyage 4 shared space allows mixing Voyage 4 family models across index/query.
store(facts: string[]): Promise<string[]>;                 // -> UUID ids
retrieve(query: string, limit?: number):                   // default RESULT_K
  Promise<{ id: string; content: string; score: number }[]>;
delete(id: string): Promise<boolean>;
```

`score` = Voyage **rerank relevance score**, NOT a probability. Higher = better. Do not threshold as a percentage.

## Validation

### `POST /facts`

Request body:

```json
{ "facts": ["..."] }
```

Rules:
- `facts` must be an array.
- Length must be `1...MAX_FACTS_PER_REQUEST`.
- Default `MAX_FACTS_PER_REQUEST=100`.
- Each fact must be a string.
- Each fact is `trim()`-normalized before storage and embedding.
- Empty facts after `trim()` are rejected.
- Each normalized fact must be at most `MAX_FACT_CHARS` characters.
- Default `MAX_FACT_CHARS=2000`.
- Duplicate normalized facts inside the same request are rejected.
- On any validation error, reject the whole request; do not partially insert.

### `POST /search`

Request body:

```json
{ "query": "...", "limit": 10 }
```

Rules:
- `query` must be a string and non-empty after `trim()`.
- Query is sent to Voyage after `trim()`.
- `limit` is optional; default is `RESULT_K`.
- If provided, `limit` must be an integer in `1...CANDIDATE_K`.
- If invalid or greater than `CANDIDATE_K`, return `400`.

### `DELETE /facts/:id`

Rules:
- `id` must be a valid UUID string.
- Invalid UUID → `400`.
- Existing id deleted → `200 { "deleted": true }`.
- Missing id → `200 { "deleted": false }`.

## Retrieval pipeline (`retrieve`)

1. `embed(query, input_type="query")` via Voyage query model.
2. pgvector ANN:
   ```sql
   SELECT id, content
   FROM knw.fact
   ORDER BY embedding <=> $1
   LIMIT CANDIDATE_K
   ```
3. Voyage rerank with original query and candidate contents.
4. Map rerank result indexes back to candidate ids.
5. Sort by rerank score desc.
6. Take `limit`.
7. Return `[{ id, content, score }]`.

`store` pipeline:

1. Validate and normalize facts.
2. Generate UUIDs in the application.
3. `embed(facts, input_type="document")` via Voyage document model.
4. Bulk insert `(id, content, embedding)`.
5. Return ids as strings.

## Health check

`GET /health` is a readiness check, not only liveness.

Behavior:
- No auth required.
- After boot schema setup, run a cheap read against `knw.fact`, for example:
  ```sql
  SELECT id FROM knw.fact LIMIT 1;
  ```
- Success → `200`.
- DB/schema failure → `503` with structured error body.

## Auth

Bearer-token middleware on all non-health API routes.

Rules:
- Compare `Authorization: Bearer <API_TOKEN>` with env `API_TOKEN` using normal string comparison.
- Missing token → `401`.
- Wrong token → `401`.
- `API_TOKEN` is required at boot.

## CORS

CORS is allowlist-based and disabled by default.

Env:

```env
CORS_ORIGINS=https://app.example.com,https://admin.example.com
```

Rules:
- If `CORS_ORIGINS` is unset or empty, do not emit CORS headers.
- If set, allow only exact origins from the comma-separated list.
- Browser preflight `OPTIONS` requests for allowed origins should be handled before bearer auth.
- Do not use wildcard `*` in v1.

## Error responses

Use structured errors:

```json
{
  "error": {
    "code": "invalid_request",
    "message": "Human readable message"
  }
}
```

Suggested status mapping:
- `400` — invalid JSON/body/params/limit/id.
- `401` — missing or wrong bearer token.
- `404` — unknown route.
- `502` — Voyage API failure.
- `503` — health readiness failure / database unavailable.
- `500` — unexpected internal error.

## Voyage integration

Direct REST (verify exact shapes against docs.voyageai.com during implementation):

- Embeddings: `POST https://api.voyageai.com/v1/embeddings`
  - body `{ input: string[], model: EMBED_MODEL|QUERY_MODEL, input_type: "document"|"query" }`
  - response contains vectors
  - `input_type` matters for asymmetric retrieval; set correctly per direction
- Rerank: `POST https://api.voyageai.com/v1/rerank`
  - body `{ query, documents: string[], model: RERANK_MODEL, top_k }`
  - response contains `[{ index, relevance_score }]`
  - map `index` back to candidate ids
- Auth header: `Authorization: Bearer $VOYAGE_API_KEY`.

No retry in v1. Surface Voyage failures as `502`.

Model defaults, covered by Voyage free-tier at the time of writing and using separate quotas for embed vs rerank:
- `EMBED_MODEL=voyage-4` — index/store documents
- `QUERY_MODEL=voyage-4-lite` — queries; shared Voyage 4 space means no re-index when swapping within Voyage 4 family
- `RERANK_MODEL=rerank-2.5-lite`

## Acceptance

- Cold boot creates `vector` extension, `knw` schema, `knw.fact` table, and HNSW index idempotently.
- `POST /facts` with N valid unique strings returns N UUID strings.
- Stored rows have non-null embeddings of dim `EMBED_DIM`.
- `POST /facts` rejects empty strings, duplicates within the request, more than 100 facts, or facts longer than 2000 chars.
- `POST /search` returns facts ordered by Voyage rerank score desc, length ≤ `limit`.
- `POST /search` rejects `limit` outside `1...CANDIDATE_K`.
- Wrong/absent bearer token → `401` on all protected routes.
- `/health` requires no auth and returns readiness based on a read from `knw.fact`.
- `DELETE /facts/:id` returns `{ deleted: true }` for existing ids and `{ deleted: false }` for missing ids.
- Invalid UUID in `DELETE /facts/:id` returns `400`.
- CORS headers are emitted only for origins in `CORS_ORIGINS`; unset env disables CORS.
- Swapping `QUERY_MODEL` between Voyage 4 family members requires **no** re-index.
- Swapping `EMBED_MODEL` to a non-compatible embedding model or changing `EMBED_DIM` requires full re-embed/re-index.
